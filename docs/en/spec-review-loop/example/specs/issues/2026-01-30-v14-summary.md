### Issue 2: Queue Failure Recovery Contradicts Confirm Plan Idempotency and State Model
- **Decision:** Accepted
- **Changes Applied:** Updated the "LLM Planner â†’ Queue Manager" cross-module interface:
  - Changed "Data passed" from `Execution Plan (status: confirmed)` to `Execution Plan (status: pending_confirmation); status transitions to confirmed only after successful enqueue acknowledgement`
  - Changed "When called" from `After user confirms plan` to `After user confirms plan (enqueue attempt)`
- **Reasoning:** The original wording created ambiguity about whether the payload's status field represented persisted state or intended transition. The fix aligns the cross-module interface with the "Inbound: Confirm Plan" contract which clearly states status only becomes `confirmed` on enqueue success.
